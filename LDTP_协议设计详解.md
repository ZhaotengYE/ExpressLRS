# 基于 LoRa 的双通道传输协议（LDTP）设计详解

> 从 ExpressLRS 的协议分析出发，到面向普通无人机场景的传输协议设计。

---

## 一、问题的起点

当我们想用 LoRa 做无人机的数据链路时，面对一个根本问题：**一条无线链路上要同时跑两种完全不同性质的数据。**

一种是**遥控数据**（RC 通道值、姿态遥测），它的特点是：高频率、低延迟、丢了没关系——因为 50ms 后更新的数据就来了，旧数据毫无价值。

另一种是**指令/命令数据**（航点上传、参数配置、MAVLink 命令），它的特点是：低频率、可以等、但绝对不能丢——你让飞机返航的命令如果丢了，后果很严重。

这两种数据对传输的要求截然相反。把它们塞进同一个协议里，是这个设计的核心挑战。

---

## 二、先看别人怎么做的：ExpressLRS 协议分析

ExpressLRS（ELRS）是目前 FPV 无人机界最流行的开源遥控链路，底层就是 LoRa。我们先拆解它的协议栈，看看它做了什么取舍。

### 2.1 ELRS 的协议分层

```
┌─────────────────────────────────────┐
│  应用层: CRSF / MAVLink / MSP       │  ← 飞控协议
├─────────────────────────────────────┤
│  可靠传输层: StubbornSender/Receiver │  ← ELRS 自己的确认机制
├─────────────────────────────────────┤
│  空中帧层: OTA_Packet (8/13 字节)    │  ← 极短的固定长度包
├─────────────────────────────────────┤
│  物理层: LoRa + FHSS 跳频           │  ← 射频
└─────────────────────────────────────┘
```

### 2.2 ELRS 的时间线：一切围绕"时隙"

ELRS 是**半双工 TDD**——同一个频率，发和收交替进行。一个硬件定时器以固定频率（比如 500Hz）产生中断，每次中断就是一个 **slot**（时隙），每个 slot 只能做一件事。

通过一个叫 TLM Ratio 的参数控制上下行比例。比如 TLM Ratio = 1:4 意味着每 4 个 slot 中，3 个给上行（地面→飞机），1 个给下行（飞机→地面）。

```
TLM Ratio = 1:4 时的时间线：

Slot 0: ──TX──→  [RC 数据]
Slot 1: ──TX──→  [MSP 分片 或 RC 数据]
Slot 2: ──TX──→  [RC 数据]
Slot 3: ←──RX──  [遥测回传]
Slot 4: ──TX──→  [RC 数据]
Slot 5: ──TX──→  [MSP 分片 或 RC 数据]
...
```

### 2.3 为 FPV 做的极端妥协：8 字节包

FPV 飞行需要极低延迟（< 5ms）和极高刷新率（250-500Hz）。在 500Hz 下，每个 slot 只有 2ms。LoRa 的 Time-on-Air 和包长度成正比，要在 2ms 内发完，包只能**非常短**。

ELRS 的标准 OTA 包只有 **8 字节**：

```
┌──────────┬────────────────────────────┬──────────┐
│ 帧头+CRC │       Payload (6 字节)      │ CRC 低位 │
│  1 byte  │                            │  1 byte  │
└──────────┴────────────────────────────┘──────────┘
```

6 字节的 payload 要放什么？RC 通道数据（4 个 10-bit 通道 + 开关位）刚好塞满。但如果要发 MSP/MAVLink 数据，扣掉分片索引等控制位，每个包只能带 **5 字节有效数据**。

一个 19 字节的 CRSF GPS 遥测帧，需要拆成 **4 个分片**，每个分片占一个 slot 来发。

### 2.4 ELRS 的"可靠"机制：StubbornSender

ELRS 为 MSP/遥测数据设计了一个叫 StubbornSender/StubbornReceiver 的机制。它的工作方式是：

**发送端**把一条完整消息拆成若干 5 字节的分片，逐个发送。每发一个分片，等对方确认。

**确认方式**是极其简陋的——整个 ACK 只有 **1 个 bit**。接收端维护一个布尔值 `telemetryConfirm`，每成功接收一个分片就翻转这个值。发送端通过观察这个 bit 是否翻转来判断对方是否收到了。

```
发送端                                  接收端
  │                                       │
  ├── 发分片 #1 ──────────────────────────→│ 收到，翻转 confirm: false→true
  │                                       │
  │←── [下一个回传包里 confirm=true] ──────┤
  │  ✓ bit 翻转了，说明 #1 被收到          │
  │                                       │
  ├── 发分片 #2 ──────────────────────────→│ 收到，翻转 confirm: true→false
  │                                       │
  │←── [回传包里 confirm=false] ───────────┤
  │  ✓ bit 又翻转了，#2 也收到了           │
  │                                       │
  ├── 发分片 #3 ──────────────────────────→│ 空中丢失！
  │                                       │
  │←── [回传包里 confirm=false] ───────────┤
  │  ✗ bit 没变！重发 #3                   │
  │                                       │
  ├── 重发分片 #3 ────────────────────────→│ 这次收到了
  │         ...                           │
```

**如果一个分片一直丢呢？** 发送端有一个 `maxWaitCount` 计数器（默认 80，动态调整），超过这个次数还没收到确认，就放弃整条消息，发一个 RESYNC 包让两端重新同步。

### 2.5 ELRS 的局限性

这个设计对 FPV 来说足够了，但有几个明显的问题：

1. **Stop-and-Wait**：一次只能传一条消息，而且卡在一个分片上就阻塞后面所有分片。分片 #3 丢了，#4、#5、#6 全部等着。

2. **没有选择性重传**：不知道具体哪个包丢了——只知道"对方没确认"。

3. **RC 和命令互斥**：一个 slot 要么发 RC 数据，要么发 MSP 分片，不能同时。有 MSP 数据在传时，RC 更新率直接减半。

4. **吞吐量极低**：5 字节/包 × 125 包/秒 ≈ 625 字节/秒。传一个 1KB 的航点列表需要 1.6 秒。

5. **包太短**：LoRa 明明可以发 200 字节的包，ELRS 只用了 8 字节。帧头占比 37.5%，极其浪费。

这些局限全是为 FPV 的"极低延迟"需求付出的代价。

## 三、重新设计：去掉 FPV 的包袱

### 3.1 场景假设

我们的目标场景是**普通无人机的控制与监控**：

- RC 控制刷新率：10-50 Hz（而不是 500 Hz）
- 需要可靠传送命令（航点、参数、返航指令）
- 需要回传遥测（GPS、姿态、电池、日志）
- 物理层：LoRa，**全双工**（上行频率 F1，下行频率 F2）
- 不需要 FHSS

全双工是一个巨大的简化——上行和下行完全独立，不需要 TDD 时隙分配，ACK 可以随时搭便车在反向的包里带回来。

### 3.2 核心设计决策

**决策 1：两种包，不混合。**

每个 LoRa 包要么是不可靠的，要么是可靠的。不像我们最初考虑过的那样把 RC 和命令塞进同一个包——那样会模糊语义。不可靠包发了就忘，可靠包有序列号、有 ACK、有重传。

**决策 2：包可以长一些。**

20Hz 的 slot 间隔是 50ms。LoRa SF7/BW250 下，200 字节的包 TOA 约 15ms，完全放得下。更长的包意味着更少的分片、更低的帧头开销比。

**决策 3：ACK 升级为 bitmap。**

ELRS 的 1-bit ACK 只能做 Stop-and-Wait。我们用 2 字节的 ACK block，包含一个累积确认号 + 7-bit 选择性确认 bitmap，可以做 Selective Repeat ARQ——丢了哪个只重传哪个。

**决策 4：ACK 搭便车。**

全双工下反方向永远有包在跑。ACK 不需要专门的包，只要在反向的任意包（不管是不可靠还是可靠）里多加 2 字节就行。

## 四、LDTP 协议详解

### 4.1 帧格式总览

协议定义了一个 Frame Control 字节作为所有帧的第一个字节：

```
Bit 7    : R    — 可靠标志 (0=不可靠, 1=可靠)
Bit 6    : A    — 是否携带 ACK block (0=无, 1=有，后面跟 2 字节 ACK)
Bit 5    : F    — 是否有更多分片 (仅 R=1 时有意义)
Bit 4-2  : TYPE — 3 bit:
                   R=0 时: 消息类型 (应用层定义，如 0=RC通道，1=GPS，2=电池...)
                   R=1 时: 消息流 ID (支持 8 路并发可靠流)
Bit 1-0  : VER  — 协议版本 (00 = v1.0)
```

### 4.2 不可靠帧（1 字节开销）

最精简的情况：

```
┌──────────────────┬─────────────────────┐
│  Frame Control   │      Payload        │
│     1 byte       │     N bytes         │
└──────────────────┴─────────────────────┘

Frame Control = [0][0][0][TYPE:3][00]
```

比如发送 RC 通道数据（TYPE=0）：Frame Control = `0b00000000` = `0x00`，后面直接跟通道数据。**1 字节开销**，ELRS 是 3 字节开销占 8 字节包——优势巨大。

如果这个包还想顺便帮对方的可靠通道带个 ACK 回去：

```
┌──────────────────┬──────────────────────┬─────────────────────┐
│  Frame Control   │  ACK Block (2 byte)  │      Payload        │
│  [0][1][0]...    │  累积确认+bitmap      │     N bytes         │
└──────────────────┴──────────────────────┴─────────────────────┘
```

多花 2 字节，把对方可靠数据的确认信息捎带回去。这个 ACK 不是为自己的数据，是帮对方的可靠通道确认。

### 4.3 可靠帧（2 字节开销）

```
┌──────────────────┬──────────────────┬─────────────────────┐
│  Frame Control   │   Seq Control    │      Payload        │
│     1 byte       │     1 byte       │     N bytes         │
└──────────────────┴──────────────────┴─────────────────────┘

Seq Control: [SEQ:6 bit][FRAG:2 bit]
```

SEQ 是序列号（0-63），FRAG 是分片索引（0-3）。

**为什么序列号 64 个就够？**

Selective Repeat ARQ 的理论要求是：序列号空间 ≥ 2 × 滑动窗口大小。我们的窗口大小推荐 8（同时最多 8 个未确认的包在飞），所以序列号空间只需 ≥ 16。64 个序列号已经非常充裕了。

**为什么分片只需要 4 个？**

假设 LoRa 包 payload 有 196 字节（200 字节包减去 4 字节最大开销），4 个分片可以传 784 字节。MAVLink v2 最大消息约 280 字节，2 个分片绑绑有余。绝大多数命令 1 个分片就搞定了。

### 4.4 ACK Block 详解（2 字节）

这是整个协议最精妙的部分。在 16 个 bit 里塞了三样东西：

```
Bit 15-13 : SID   — 被确认的消息流 ID (3 bit)
Bit 12-7  : CACK  — 累积确认号 (6 bit)
Bit 6-0   : SACK  — 选择性确认 bitmap (7 bit)
```

**CACK（Cumulative ACK）**：表示"这个序列号以及之前的所有包，我都收到了"。类似 TCP 的 ACK 号。

**SACK（Selective ACK）**：从 CACK+1 开始的 7 个位，每位表示对应序列号是否收到。

#### 示例

发送端发了 SEQ 10、11、12、13、14、15。接收端收到了 10、11、13、15（12 和 14 丢了）。

```
CACK = 11（11 及之前全部收到）

SACK bitmap（从 CACK+1=12 开始）：
  Bit 6 → SEQ 12: 0 ← 没收到
  Bit 5 → SEQ 13: 1 ← 收到了
  Bit 4 → SEQ 14: 0 ← 没收到
  Bit 3 → SEQ 15: 1 ← 收到了
  Bit 2 → SEQ 16: 0
  Bit 1 → SEQ 17: 0
  Bit 0 → SEQ 18: 0

SACK = 0b0101000
```

发送端看到这个 ACK 后：只需要重传 SEQ 12 和 SEQ 14。其他的都不用动。

#### 对比 ELRS

| | ELRS | LDTP |
|--|------|------|
| ACK 信息量 | 1 bit（翻转/不翻转） | 16 bit（精确到每个 SEQ） |
| 丢包处理 | 卡住，重发当前分片 | 只重发丢的那个 |
| 一次能确认几个包 | 1 个 | 最多 8 个 |
| 需要额外的 ACK 包 | 不需要（但效率低） | 通常不需要（搭便车） |

### 4.5 重传机制

和 ELRS 的"数计数器超时就放弃"不同，LDTP 使用**自适应超时 + 指数退避**：

```
初始状态（无 RTT 样本时）：
  RTO = RTO_INITIAL = 4 ticks (200ms @ 20Hz)

收到第一个 RTT 样本后（Jacobson/Karels 算法）：
  RTT_est = sample
  RTT_dev = sample / 2
  RTO = clamp(RTT_est + 4 × RTT_dev, RTO_MIN, RTO_MAX)

后续 RTT 更新：
  err     = sample − RTT_est
  RTT_est = 0.875 × RTT_est + 0.125 × sample    ← 指数移动平均
  RTT_dev = 0.75  × RTT_dev + 0.25  × |err|      ← 偏差估计
  RTO     = clamp(RTT_est + 4 × RTT_dev, RTO_MIN, RTO_MAX)

Karn 算法：只从未重传过的包采样 RTT（重传包的 ACK 有歧义）

重传策略：
  每次发送：timer = min(RTO_MAX, RTO × 2^min(重传次数, 4))
  最多重传 5 次，之后通知应用层"送达失败"

常量：RTO_MIN=2 ticks, RTO_MAX=100 ticks, RTO_MAX_EXP=4
```

在 20Hz 的链路上，RTT ≈ 2 ticks（发一个包，下一个反向包就能带 ACK 回来），自适应 RTO 会收敛到约 4 ticks。信号差时 RTO 自动增大避免无效重传，信号好时缩小以加快恢复。

### 4.6 主动 ACK 帧机制

全双工下 ACK 通常搭便车在反向的包里，但如果反向没有数据要发（或反向的包一直丢），ACK 就一直送不回去，导致发送端不必要地重传甚至认为送达失败——这就是 **ACK 饥饿** 问题。

解决方案：接收端维护 `ack_pending_since` 时间戳。当收到可靠帧后，如果在 `ACK_DELAY_TICKS`（推荐 2 ticks = 100ms）内没有搭便车机会，就主动发送一个 **ACK-only 帧**（仅 3 字节）。

```
接收端逻辑：
  收到可靠帧 → 设置 ack_pending_since = 当前 tick
  每次发送带 ACK 的包 → 重置 ack_pending_since = -1
  每个 tick 检查：
    if ack_pending_since >= 0 且 (now - ack_pending_since) >= ACK_DELAY_TICKS:
      发送 ACK-only 帧
```

这个机制确保 ACK 在最坏情况下也能在 2 个 tick（100ms）内送达。

### 4.7 链路质量监控

协议提供一个面向应用层的**链路质量指标**，基于滑动窗口内的包送达率：

```
滑动窗口大小：50 个包（20Hz 下覆盖 2.5 秒）
每发送一个包，记录送达/丢失结果
quality = 窗口内送达包数 / 窗口大小 × 100%
```

应用层可以根据链路质量做出决策：

| 质量等级 | 范围 | 建议动作 |
|----------|------|----------|
| 优秀 | 80-100% | 正常运行 |
| 边缘 | 50-79% | 警告用户，考虑增加重传次数 |
| 差 | 20-49% | 触发安全警告，仅传输必要数据 |
| 危险 | 0-19% | 触发返航或紧急程序 |

在 Simulation 界面上，链路质量实时显示在 GCS 和 UAV 节点下方的进度条和 overlay 数值中。

## 五、一个完整的工作场景

假设 20Hz 上行、20Hz 下行，地面站正在给飞机上传一个 300 字节的航点列表（需要 2 个分片）。

```
时间    上行 (GCS → UAV)                     下行 (UAV → GCS)
────────────────────────────────────────────────────────────────
  0ms   [不可靠] RC 通道数据                  [不可靠] 姿态数据
                                              （无 ACK，因为上行还没有可靠数据）

 50ms   [可靠] 航点列表 SEQ=5 FRAG=0          [不可靠] GPS 数据
        (第一片, 196字节)                      （无 ACK，上行可靠数据刚发出）

100ms   [可靠] 航点列表 SEQ=5 FRAG=1          [不可靠] 电池数据 + ACK(CACK=5, SACK=0)
        (第二片,剩余104字节,最后一片)            ← 搭便车确认：SEQ 5 的 FRAG 0 收到了

150ms   [不可靠] RC 通道数据                  [不可靠] 姿态 + ACK(CACK=5, SACK全1)
                                              ← FRAG 1 也收到了，航点上传完成！

200ms   [不可靠] RC 通道数据                  [不可靠] GPS 数据
        （一切恢复正常，无可靠数据在传）
```

从发出到确认收到，整个航点上传在 **150ms 内完成**（3 个 slot）。

如果第一个分片丢了呢？

```
 50ms   [可靠] SEQ=5 FRAG=0 ──→ 丢失!        [不可靠] GPS
100ms   [可靠] SEQ=5 FRAG=1 ──→ 收到          [不可靠] 电池（无 ACK，FRAG 0 没收到）
150ms   [不可靠] RC 通道                      [不可靠] 姿态 + ACK(CACK=4, SACK=bit5=1)
                                              ← 告诉发送端："4之前都收到了，
                                                 SEQ 5 FRAG 1 收到了，但 FRAG 0 没有"

200ms   RTO 到期，发送端知道 FRAG 0 丢了
        → 只重传 FRAG 0

250ms   [可靠] SEQ=5 FRAG=0 (重传) ──→ 收到   [不可靠] GPS + ACK(CACK=5)
                                              ← 全部收到！航点上传完成
```

重传只针对丢失的那一个分片，不影响其他任何数据。这就是 Selective Repeat 的威力。

## 六、和 ELRS 方案的对比总结

| 维度 | ELRS | LDTP |
|------|------|------|
| 双工方式 | 半双工 TDD | 全双工 |
| 包大小 | 8-13 字节 | 灵活，推荐 64-200 字节 |
| 帧头开销 | 3 字节 / 8 字节 = 37.5% | 1-4 字节 / 200 字节 = 0.5-2% |
| ACK 方式 | 1 bit 翻转 | 2 字节 (累积ACK + 7bit bitmap) |
| 重传策略 | Stop-and-Wait，卡住就阻塞 | Selective Repeat，只重传丢的 |
| 并发可靠流 | 1 条 | 最多 8 条 |
| RC 和命令 | 互斥，不能同时发 | 各自独立的包，互不影响 |
| 分片传输 300 字节 | ~60 个分片 × 60 个 slot | 2 个分片 × 2 个 slot |
| 送达通知 | 无 (内部消化) | callback 通知应用层 |
| 超时机制 | 固定计数器 | 自适应 RTO + 指数退避（Jacobson/Karels） |
| ACK 保障 | 依赖下一个回传 slot | 搭便车优先 + 主动 ACK-only 帧兜底 |
| 链路监控 | 无 | 滑动窗口链路质量指标（应用层可见） |
| 适合场景 | FPV (极低延迟) | 普通无人机 (可靠+灵活) |

## 七、协议线格式速查

```
不可靠帧，无 ACK（1 字节开销）：
  [0|0|0|TYPE(3)|VER(2)]  [Payload...]

不可靠帧，搭载 ACK（3 字节开销）：
  [0|1|0|TYPE(3)|VER(2)]  [SID(3)|CACK(5)][CACK(1)|SACK(7)]  [Payload...]

可靠帧，无 ACK（2 字节开销）：
  [1|0|F|STREAM(3)|VER(2)]  [SEQ(6)|FRAG(2)]  [Payload...]

可靠帧，搭载 ACK（4 字节开销）：
  [1|1|F|STREAM(3)|VER(2)]  [SID(3)|CACK(5)][CACK(1)|SACK(7)]  [SEQ(6)|FRAG(2)]  [Payload...]

纯 ACK 帧（3 字节，无 Payload）：
  [0|1|0|111|VER(2)]  [SID(3)|CACK(5)][CACK(1)|SACK(7)]
```
