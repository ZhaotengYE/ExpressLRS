Key Findings & Protocol Assessment

Strengths

STRENGTH 100% delivery reliability up to ~50% uniform packet loss (Monte Carlo confirmed).
STRENGTH Selective Repeat with SACK reduces retransmissions by ~50% compared to Go-Back-N (13 vs ~24 for 20 messages at 30% loss).
STRENGTH Burst loss resilience — even with 90% loss in "bad" state (Gilbert-Elliott), all messages delivered thanks to retransmission backoff naturally waiting out bad periods.
STRENGTH Piggybacked ACKs on both reliable and unreliable frames enable bidirectional reliable traffic without extra overhead.
STRENGTH Sequence wraparound works flawlessly: 70 messages through 64-number space with no issues.
STRENGTH Blackout recovery: messages queued during total blackout are delivered once channel returns, with latency ≈ blackout duration.

Limitations

LIMITATION ACK starvation: If the reverse channel is 100% dead, sender declares failure even though data was received. This is inherent to ALL ARQ protocols — no fix possible without a separate ACK channel.
LIMITATION Failure threshold ~60% loss: Delivery probability drops below 100% at ~60% bidirectional loss. At 90%, only ~45-50% of messages succeed with MAX_RETRIES=5. Increasing MAX_RETRIES would help but extend failure detection time.
ISSUE Slow failure detection: Exponential backoff means MAX_RETRIES=5 takes 9.2 seconds to fully exhaust. A stalled message occupies a window slot for this entire period, reducing effective throughput.
ISSUE Latency explosion under high loss: At 50%+ loss, max latency jumps to 3+ seconds (61+ ticks). For safety-critical commands like RTH, this may be unacceptable.
INFO ACK-only frames unused: The protocol defines build_ack_only() but never needs it in the simulation because background traffic (RC/telemetry every tick) always provides piggybacking opportunities. Applications without background traffic would need explicit ACK generation logic.

Recommendations

RECOMMEND Priority-based retransmission: Safety-critical messages (RTH, EMERGENCY) should use aggressive retransmission (shorter initial RTO, no backoff cap) to minimize latency at the cost of more channel usage.
RECOMMEND Adaptive MAX_RETRIES: Instead of fixed 5, base it on channel quality estimation. In good conditions, 3 retries suffice; in poor conditions, allow more but with shorter backoff.
RECOMMEND Standalone ACK trigger: If no reverse-direction data for N ticks, automatically send an ACK-only frame. This protects against asymmetric traffic patterns.
RECOMMEND Link quality monitoring: Track delivery/ACK success ratio to provide an application-level "link health" metric, enabling the flight controller to make informed decisions (e.g., auto-RTH when link degrades).
